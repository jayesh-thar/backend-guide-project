Before writing a single line of code — a full auth flow diagram was designed.
This went through 6 iterations, each improving the previous version.

Final Flow Agreed Upon:

  REGISTER:
    User sends data
        → check email already exists?
            YES → "Email already registered" error
            NO  → hash password → save to DB → JWT token
               → store in cookie → send welcome email → success

  LOGIN:
    User sends email + password
        → check user exists in DB?
            NO  → "User not found" error
            YES → bcryptjs.compare(password, hashedPassword)
                WRONG → "Invalid password" error
                RIGHT → JWT token → store in cookie → success

  LOGOUT:
    User hits /logout
        → clear cookie (maxAge: 0) → success

  FORGOT PASSWORD:
    User sends email
        → check user exists?
            NO  → "Email not found" error
            YES → generate reset token → save token + expiry to DB
               → send reset email with link → success

  RESET PASSWORD:
    User sends token (from URL) + new password
        → find user by token → check token not expired
            INVALID/EXPIRED → error
            VALID → hash new password → save → clear token → success

  PROTECTED ROUTE:
    Any request to protected endpoint
        → middleware checks cookie for JWT
            NO TOKEN   → 401 Unauthorized
            BAD TOKEN  → 401 Unauthorized
            GOOD TOKEN → decode → attach userId to req → continue


Key Doubts Raised During Diagram Phase:
  Q: Where does JWT go after login?
  A: Stored in an httpOnly cookie — browser auto sends it with every request

  Q: What is httpOnly and why?
  A: JavaScript cannot read httpOnly cookies — protects against XSS attacks

  Q: Why cookie and not localStorage?
  A: Cookie auto-sends with requests. localStorage is vulnerable to XSS.

  Q: What is a protected route vs normal route?
  A: Public = anyone can access. Protected = must be logged in (valid token required)

Concepts Learned:

  1. import vs require
     ┌─────────────────────────────────────────────────────┐
     │ CommonJS (old)     │ ES Module (new)                │
     │ require('x')       │ import x from 'x'             │
     │ module.exports = x │ export default x              │
     │ exports { x, y }   │ export { x, y }               │
     └─────────────────────────────────────────────────────┘

  2. export default vs named export
     - export default  → one main thing per file → import x from '...'
     - export { x, y } → multiple things → import { x, y } from '...'
     - Named imports must match export name exactly

  3. dotenv — why and how
     - .env file stores secrets (DB password, JWT secret)
     - dotenv.config() reads .env → loads into process.env
     - Must be called BEFORE anything that uses process.env
     - .env values accessed via process.env.VARIABLE_NAME
     - .env only supports # comments, NOT // comments

  4. app.listen() — what it needs
     - First param: port number
     - Second param: callback function (runs when server starts)
     - process.env.PORT || 5000 — fallback if .env not loaded

  5. ES Module import hoisting issue
     - ALL import statements are hoisted before any code runs
     - Even dotenv.config() runs AFTER imports
     - Solution: use  import 'dotenv/config'  instead
     - This guarantees .env is loaded before anything

      try {

  Why try/catch:
    - DB calls can fail anytime
    - findOne() / create() / save() → all can throw errors
    - Without catch → server crashes with no response
    - With catch → graceful 500 error sent back

  Why return before res.json():
    - Without return → function keeps running after sending response
    - Causes: "Cannot set headers after they are sent" error
    - Always return when sending early error response

  Response Format (Production Standard):
    res.status(200).json({
      success: true/false,
      message: 'Human readable message',
      data: {...}  // optional
    })

  register() Flow Built:
    1. const { username, email, password } = req.body
    2. if (!username || !email || !password) → 400 error
    3. if (password.length < 6) → 400 error
    4. const existingUser = await User.findOne({ email })
    5. if (existingUser) → 400 "User already exists"
    6. const hashedPassword = await bcryptjs.hash(password, 10)
    7. const user = await User.create({ username, email, password: hashedPassword })
    8. const userResponse = await User.findById(user._id).select('-password')
    9. res.status(201).json({ success: true, message: '...', userResponse })

  login() Flow Built:
    1. const { email, password } = req.body
    2. if (!email || !password) → 400 error
    3. const userExisted = await User.findOne({ email })

    5. const correctPassword = await bcryptjs.compare(password, userExisted.password)
    6. if (!correctPassword) → 400 "Incorrect password"

    8. res.status(200).json({ success: true, message: '...' })

    1. res.cookie('token', '', { maxAge: 0 })
    2. res.status(200).json({ success: true, message: 'Logged out' })

                    BCRYPT DEEP DIVE — KEY UNDERSTANDING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This was a major doubt that was fully resolved.

  Q: Can I hash the login password and compare with DB hash?
  A: NO — bcrypt adds RANDOM salt every time. Same password → different hash always.

  SHA256 vs bcrypt:
  ┌────────────────────┬────────────────────┬────────────────────┐
  │                    │ SHA256             │ bcrypt             │
  ├────────────────────┼────────────────────┼────────────────────┤
  │ Salt               │ None               │ Random every time  │
  │ Same input = same  │ Yes (deterministic)│ No (always diff)   │
  │ Reversible         │ No                 │ No                 │
  │ Speed              │ Very fast          │ Slow (intentional) │
  │ Rainbow table safe │ No (vulnerable)    │ Yes (protected)    │
  │ For passwords      │ No                 │ Yes                │
  └────────────────────┴────────────────────┴────────────────────┘

  How bcrypt.compare() works internally:
    1. Extract salt from stored hash (salt is embedded inside the hash string)
    2. Extract rounds from stored hash
    3. Hash the plain password using SAME salt + SAME rounds
    4. Compare new hash with stored hash
    5. Match → true | No match → false

  Hash string structure:
    $2a$10$N9qo8uLOickgx2ZMRZoMye.restofhash
     ↑   ↑  ↑_____________________↑↑_________↑
     ver rounds    salt (22 chars)    actual hash

  Who decides the salt?
    → bcrypt generates it internally and randomly every time
    → You only control rounds (10 is production standard)
    → Higher rounds = slower = harder to brute force

  Proof ran mentally:
    bcrypt.hash("hello123", 10) → "$2a$10$ABC..."
    bcrypt.hash("hello123", 10) → "$2a$10$XYZ..."  (different!)
    SHA256("hello123")          → "0b8da7b3..."  (same always)
    SHA256("hello123")          → "0b8da7b3..."  (same always)

─────────────────────────────────────
JWT — JSON Web Token
─────────────────────────────────────

  Why JWT?
    HTTP is stateless — server forgets you after every request.
    JWT solves this: after login, server creates a token with your userID.
    Every future request sends this token → server knows who you are.

  JWT Structure (3 parts separated by dots):
    eyJhbGciOiJIUzI1NiJ9.eyJ1c2VySUQiOiIxMjMifQ.abc123
            ↑                      ↑                  ↑
        Header               Payload              Signature
      (algorithm)          (userID+exp)         (secret stamp)

  IMPORTANT: Payload is encoded NOT encrypted — anyone can read it.
  → Never store passwords or sensitive data in JWT payload.

  jwt.sign() — Creating a token:
    jwt.sign(
      { userID },              → payload (what to store inside)
      process.env.JWT_SECRET,  → secret key (server's stamp)
      { expiresIn: '7d' }      → options (when it expires)
    )

  jwt.verify() — Verifying a token:
    const decoded = jwt.verify(token, process.env.JWT_SECRET)
    → checks signature is valid (not tampered)
    → checks token not expired
    → returns payload if valid
    → throws error if invalid → caught by catch block

  JWT Secret Key:
    → 256 bit (64 chars) is production standard
    → Generated using online JWT secret generator
    → Stored in .env — never hardcode

  Why new token every login?
    Token contains expiry timestamp — always different → always different token.
    This is normal and correct behavior (stateless approach).

COOKIES
─────────────────────────────────────

  What is a cookie?
    Small data browser stores and automatically sends with every request
    to same server. We store JWT inside cookie.

  Cookie vs localStorage:
  ┌────────────────────┬───────────────────┬───────────────────┐
  │                    │ Cookie            │ localStorage      │
  ├────────────────────┼───────────────────┼───────────────────┤
  │ Auto sent          │ Yes               │ No (manual)       │
  │ httpOnly option    │ Yes               │ No                │
  │ XSS safe           │ Yes (httpOnly)    │ No (vulnerable)   │
  │ Expiry control     │ Yes               │ Manual            │
  │ Best for           │ Auth tokens       │ Non-sensitive     │
  └────────────────────┴───────────────────┴───────────────────┘

  Cookie options used:
    httpOnly: true      → JS cannot read cookie (XSS protection)
    secure: true        → HTTPS only
    sameSite: 'strict'  → same site only (CSRF protection)
    maxAge: 7*24*60*60*1000 → 7 days in milliseconds

  Cookie Lifecycle:
    LOGIN  → res.cookie('token', jwtString, {...}) → browser stores
    REQUEST → browser auto sends cookie → server reads req.cookies.token
    LOGOUT → res.cookie('token', '', { maxAge: 0 }) → browser removes

  Why cookie-parser middleware?
    Without it: req.cookies = undefined
    With it: req.cookies.token = your JWT string

─────────────────────────────────────
generateToken.js
─────────────────────────────────────

  Why separate file?
    Both register and login need to create tokens.
    DRY principle — Don't Repeat Yourself.
    One reusable function used everywhere.

  Function signature:
    const generateToken = (userID, res) => { ... }
    → userID: stored inside token payload
    → res: needed to call res.cookie() to set the cookie

  Why return token?
    → Mobile apps don't use cookies — they need token in response body
    → Returning it gives flexibility for both web and mobile

─────────────────────────────────────
AUTH MIDDLEWARE (auth.middleware.js)
─────────────────────────────────────

  What is middleware?
    Function that runs BETWEEN request and controller.
    Has access to req, res, and next.
    next() → passes control to next function in chain.
    Without calling next() → request stops at middleware forever.

  verifyToken flow:
    1. const token = req.cookies.token
    2. if (!token) → return 401 "No token"
    3. const decoded = jwt.verify(token, JWT_SECRET)
    4. req.userId = decoded.userID
    5. next()
    catch → 401 "Invalid token"

  How to use on routes:
    router.get('/profile', verifyToken, controller)
    → verifyToken runs first → if valid → controller runs
    → if invalid → 401 returned, controller never runs

  Protected vs Public routes:
    Public:    router.post('/register', register)
    Protected: router.get('/profile', verifyToken, profileController)

  Profile test route added:
    router.get('/profile', verifyToken, (req, res) => {
      res.json({ success: true, userId: req.userId })
    })

    ✅ GET /profile with valid cookie → { success: true, userId: '...' }

HOW EMAIL WORKS
─────────────────────────────────────

  Full email flow:
    Your Code (nodemailer)
        → SMTP Server (Mailtrap sandbox)
        → (caught here — never delivered in sandbox)
        → In production: real SMTP → user's inbox

  SMTP = Simple Mail Transfer Protocol
    → Language email servers use to communicate
    → Like HTTP is for web, SMTP is for email
    → Ports: 25, 465, 587, 2525 (different security levels)
    → Port 2525 with secure: false → STARTTLS (what Mailtrap uses)

  Nodemailer:
    → Just a Node.js library that knows how to talk to SMTP servers
    → Not a mail server — just a messenger

  Mailtrap Sandbox:
    → Catches ALL emails — never delivers to real users
    → Shows in Mailtrap dashboard
    → Perfect for development testing

  Transporter:
    → Configured connection to SMTP server
    → Set up once → use everywhere to send emails

CRITICAL BUG — dotenv + ES Modules
─────────────────────────────────────

  Problem:
    Transporter was created at file import time.
    But ES Modules hoist ALL imports before any code.
    So process.env values were undefined when mailtrap.js loaded.
    Result: nodemailer fell back to localhost:587 (default)
    Error: "ECONNREFUSED 127.0.0.1:587"

  Solution 1: Use import 'dotenv/config' in server.js
  Solution 2: Wrap transporter in a factory function

  Factory function approach:
    // mailtrap.js
    const createTransporter = () => {
      return nodemailer.createTransport({...})
    }
    export default createTransporter

    // sendEmail.js
    const transporter = createTransporter()  // created at call time
    → By call time, dotenv is fully loaded ✅

  Why this works:
    Transporter is now created WHEN sendEmail() is called
    Not when file is imported
    By that time all .env values are loaded

─────────────────────────────────────
Why Rate Limiting?
─────────────────────────────────────

  Without it:
    → Attacker sends 1000 login attempts → brute force password
    → Attacker sends 10,000 requests → server crashes (DDoS)

  Types of limiters needed:
    generalLimiter → whole app → 100 req/15min → normal browsing
    authLimiter    → auth routes → 10 req/15min → strict for security

  Why two limiters? (Important question asked):
    Q: Why not just one limiter for everything?
    A: If max=10 for everything → normal user gets blocked while browsing
       Normal web page can easily use 10+ requests just loading assets.
       Auth needs strict limit. Regular browsing needs generous limit.

  Building analogy:
    generalLimiter = building entrance → 100 people/hour
    authLimiter    = server room → 10 people only (sensitive area)

─────────────────────────────────────
How Rate Limiter Works Internally
─────────────────────────────────────

  How it knows same IP:
    → Every request has req.ip (automatically available)
    → Stores counter per IP in memory:
      { "192.168.1.1": { count: 3, resetTime: ... } }
    → Same IP → find in memory → increment → check limit
    → Default storage: memory (resets on server restart)
    → Production: use Redis for persistent storage

  How it blocks without explicit next():
    → rateLimit() creates middleware with next() built inside
    → Under limit → calls next() internally → router runs
    → Over limit → calls res.json(message) → router never runs

  standardHeaders vs legacyHeaders:
    standardHeaders: true  → sends new RateLimit-* headers
    legacyHeaders: false   → disables old X-RateLimit-* headers
    → Headers tell client: requests remaining, reset time
    → Frontend can show: "3 attempts remaining"

  authLimiter config:
    const authLimiter = rateLimit({
      windowMs: 15 * 60 * 1000,   → 15 minutes
      max: 10,                     → 10 attempts
      message: { success: false, message: "Too many attempts..." },
      standardHeaders: true,
      legacyHeaders: false,
    })

  Applied to routes:
    app.use('/api/auth', authLimiter, authRouter)
    → Same path, multiple middleware — runs in order left to right

  Middleware execution order (full chain):
    express.json()
        ↓
    cookieParser()
        ↓
    authLimiter (check rate limit)
        ↓
    authRouter (find matching route)
        ↓
    verifyToken* (*protected routes only)
        ↓
    Controller (run logic)
        ↓
    Response sent
